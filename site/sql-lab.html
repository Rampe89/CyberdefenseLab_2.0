SQL-Lab – vom normalen Suchen zur manipulierten Abfrage

Dieses Lab simuliert keine „coole Hacker-Magie“, sondern ein sehr nüchternes Problem: Eine Anwendung übernimmt Eingaben ungeprüft in eine Datenbankabfrage. Genau deshalb ist SQL-Injection didaktisch so wertvoll: Man sieht sofort, wie kleine Eingaben große Wirkung entfalten können.

Lernweg

Du gehst hier bewusst in vier klaren Schritten vor:

  1. Beobachten: Wie funktioniert eine normale Suche?
  2. Ausprobieren: Was passiert, wenn die Eingabe die Logik verändert?
  3. Verstehen: Warum liefert die Abfrage plötzlich alle Datensätze?
  4. Absichern: Warum stoppt Parameterbindung genau diesen Effekt?

Mini-Glossar

Datenbank: strukturierte Sammlung von Daten.

SQL: Sprache für Datenbankabfragen.

WHERE: legt fest, welche Zeilen ausgewählt werden.

Parameterbindung: Eingaben bleiben Werte und werden nicht Teil der Befehlsstruktur.

Phase 1 – Normale Suche
Ziel: normale WHERE-Abfrage lesen und prüfen

Wähle einen Namen aus der Tabelle, zum Beispiel alice. Die Anwendung soll genau einen Datensatz finden.

Gebildete Abfrage 
SELECT * FROM students WHERE username = 'alice';
Passt. Eine normale Suche liefert genau den passenden Datensatz.
Phase 2 – Manipulierte Eingabe
Ziel: Wirkung einer unsicheren String-Verkettung sehen

Jetzt gibst du bewusst keinen normalen Namen ein, sondern eine Eingabe, die die Logik der Abfrage verändert.

Gebildete Query 
SELECT * FROM students WHERE username = '...';

Hier wird die Eingabe direkt in den SQL-Text eingebaut. Genau das ist der kritische Fehler.

Treffer. Wenn alle Zeilen erscheinen, wurde die Bedingung effektiv ausgehebelt.
Phase 3 – Warum funktioniert das?
Ziel: nicht nur Effekt sehen, sondern Ursache benennen

Was die Anwendung wollte

SELECT * FROM students
WHERE username = 'alice';

Die WHERE-Bedingung sollte nur einen passenden Datensatz auswählen.

Was tatsächlich passiert

SELECT * FROM students
WHERE username = '' OR '1'='1' --';

Der eingeschleuste Teil macht die Bedingung immer wahr. Der Rest wird auskommentiert.

Didaktischer Kern:
  • Das Problem ist nicht „zu viel Technik“, sondern ein Denkfehler im Umgang mit Eingaben.
  • Die Anwendung behandelt Nutzereingaben fälschlich wie Bestandteile eines Befehls.
  • Wer das versteht, versteht auch viele andere Sicherheitsprobleme besser.
Phase 4 – Sichere Variante
Ziel: denselben Input mit Parameterbindung testen

Jetzt wird dieselbe Eingabe noch einmal verwendet. Der Unterschied: Die Anwendung trennt SQL-Struktur und Eingabewert.

Unsicher

SELECT * FROM students
WHERE username = 'EINGABE';

Die Eingabe landet direkt im SQL-Text.

Sicher

SELECT * FROM students
WHERE username = ?
-- bind: ['EINGABE']

Die Eingabe bleibt ein Wert. Die Struktur der Abfrage bleibt stabil.

Gebildete Query 
SELECT * FROM students WHERE username = ?  -- bind: ['...']

Merksatz: Eingaben sind Daten, nicht Befehle.

Genau das ist der Punkt. Die gleiche Eingabe führt hier nicht mehr dazu, dass alle Datensätze erscheinen.

Worauf man achten soll

  • Welche Eingabe erzeugt einen normalen Treffer?
  • Welche Eingabe verändert plötzlich die Logik?
  • Wann sieht man die eigentliche Ursache in der Query?
  • Warum löst dieselbe Eingabe im sicheren Modus keinen Massen-Treffer mehr aus?

Kurz-Check

Der Check bewertet nicht deine Formulierungen, sondern ob du die entscheidenden Schritte praktisch durchgeführt hast.

Noch keine Auswertung.

Aufgaben für den Unterricht

  1. Notiere die Eingabe, die im unsicheren Modus alle Zeilen sichtbar macht.
  2. Beschreibe in einem Satz, warum das funktioniert.
  3. Vergleiche unsichere und sichere Query sprachlich miteinander.
  4. Nenne zwei Schutzmaßnahmen, die über dieses Lab hinaus wichtig sind.
