Mini-Glossar

Datenbank: eine strukturierte Sammlung von Daten, zum Beispiel zu Schüler:innen.

DBMS: das Programm, das diese Daten verwaltet, etwa MySQL, PostgreSQL oder SQLite.

SQL: die Sprache, mit der Daten abgefragt oder verändert werden.

Tabelle: Daten in Zeilen und Spalten.

WHERE: der Teil einer Abfrage, der eine Bedingung setzt.

  1. normale Suche durchführen,
  2. Manipulation beobachten,
  3. Ursache sprachlich erklären,
  4. sichere Lösung vergleichen.

Phase 1: Normale Abfrage

Suche dir einen Namen aus der Tabelle. Beispiel: alice.

Abfrage in Worten: Hole alle Spalten aus students, aber nur die Zeile, deren username gleich alice ist.

SELECT * FROM students WHERE username = 'alice';
Gut. Die normale Abfrage liefert genau einen passenden Datensatz.

Phase 2: Manipulation ausprobieren

Jetzt wird absichtlich kein echter Name eingegeben. Beispiel: ' OR '1'='1' --.

Generierte Query (unsicher) 
SELECT * FROM students WHERE username = '...';

Hier wird die Eingabe direkt in die Zeichenkette eingebaut. Genau das macht die Manipulation möglich.

Richtig. Alle Zeilen wurden angezeigt. Genau das ist die typische Wirkung der Injection in dieser Simulation.

Phase 3: Warum funktioniert das?

  1. Die Anwendung baut den SQL-Text, indem sie die Eingabe einfach hineinschreibt.
  2. Die Eingabe fügt eine neue Bedingung ein, zum Beispiel OR '1'='1'.
  3. Diese Bedingung ist immer wahr. Deshalb werden alle Zeilen zurückgegeben.
Manipulierte Abfrage 
SELECT * FROM students WHERE username = '' OR '1'='1';

Der markierte Teil stammt aus der Eingabe und verändert die Logik der Abfrage.

Phase 4: Sichere Variante (Parameterbindung)

Die gleiche Eingabe wird jetzt noch einmal verwendet. Diesmal bleibt die Struktur der Abfrage unverändert, weil die Eingabe nur als Wert gebunden wird.

Unsicher

Eingabe wird in den SQL-Text eingesetzt.

SQL-Text: SELECT * FROM students WHERE username = 'EINGABE';

Die Eingabe kann die Logik der Abfrage verändern.

wird eingebaut
Sicher

Eingabe und SQL-Struktur bleiben getrennt.

SQL-Text: SELECT * FROM students WHERE username = ?
bind: ['EINGABE']

Die Eingabe bleibt Datenwert und wird nicht zu SQL-Logik.

Merksatz: Eingaben sind Daten, nie Befehle.

Bei der sicheren Variante bleibt der SQL-Text gleich. Nur der gebundene Wert ändert sich.

Generierte Query (sicher) 
SELECT * FROM students WHERE username = ?  -- bind: ['...']

Die Abfrage bleibt unverändert. Die Eingabe wird nur als Parameter gebunden.

Gut. Im sicheren Modus erscheinen nicht alle Zeilen, obwohl dieselbe Eingabe verwendet wurde.

Aufgaben

  1. Notiere die Eingabe, die in Phase 2 alle Zeilen zeigt.
  2. Erkläre in einem Satz, warum diese Eingabe wirkt.
  3. Beschreibe, was sich in Phase 4 ändert.
  4. Nenne zwei Schutzmaßnahmen gegen SQL Injection.
